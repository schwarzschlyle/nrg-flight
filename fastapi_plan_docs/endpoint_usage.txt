1. Authentication Flow
Register page

On form submit → POST /auth/register

Login page

On form submit → POST /auth/login
Store the returned JWT access token and refresh token (memory + httpOnly cookie respectively)

Every subsequent request

Attach Authorization: Bearer <token> header automatically via an HTTP interceptor/middleware

Token expiry

When any request returns 401 → silently call POST /auth/refresh to get a new access token, then retry the original request

Logout button

Call POST /auth/logout, then clear tokens from client storage

"Welcome, [name]" header display

On app load after login → GET /auth/me to hydrate the current user's name and profile into your app state


2. Date & Time Slot Selection Page
This is the first page a user lands on after logging in.
On page load (with a default date pre-selected, e.g. today)

Call GET /slots?date=YYYY-MM-DD
This returns all 24 time slots in one shot with their associated flight number, gate, and available seat count
Render 24 slot cards/buttons in the UI

When user changes the date picker

Call GET /slots?date=YYYY-MM-DD again with the new date
Re-render the 24 slot cards with fresh data

When user clicks a time slot card

Call GET /slots/{slot_id}/flight?date=YYYY-MM-DD
This returns the definitive flight_id, flight_number, and gate for that slot
Store this in your routing state, then redirect the user to the seat picker page with the flight_id in the URL (e.g. /flights/{flight_id}/seats)


3. Seat Picker Page
User arrives here after choosing a time slot. The flight_id is in the URL.
On page load

Call GET /flights/{flight_id} to display flight info at the top of the page (flight number, gate, departure time, aircraft model)
Call GET /flights/{flight_id}/seats to fetch the full seat map
Render the seat grid. For every seat where is_booked = TRUE, render the button as disabled/greyed. For is_booked = FALSE, render it as selectable.

"Assign first available seat" feature (requirement 02)

When user clicks an "Auto-assign" or "Pick for me" button → call GET /flights/{flight_id}/seats/first-available
Pre-select (highlight) the returned seat in the UI
User then clicks Book to confirm

"Assign a specific seat" feature (requirement 03)

When user clicks a specific seat button in the grid, that seat becomes selected in local UI state — no API call needed here since you already have the full seat map loaded
Optionally, if you support a manual seat code text input (e.g. type "12C") → call GET /flights/{flight_id}/seats/{seat_code} to validate it exists and is still available before allowing the user to proceed

Real-time availability (optional but recommended)

Poll GET /flights/{flight_id}/seats every 15–30 seconds, or use a WebSocket, to refresh is_booked states while the user is on the page — so a seat doesn't appear available if another user just booked it


4. Booking Confirmation
User has selected a seat and clicks the Book button.

Call POST /bookings with { flight_id, seat_id }
On 201 success → redirect to a booking confirmation page, show flight number, gate, seat code, and booking ID
On 409 Conflict → the seat was just taken by someone else. Re-call GET /flights/{flight_id}/seats to refresh the seat map and show an error ("This seat was just booked. Please choose another.")
On 401 → token expired, trigger the refresh flow then retry


5. My Bookings Page
On page load

Call GET /bookings (default: all bookings, paginated)
Render a list of the user's bookings with flight info, seat, and status

Filter by status (e.g. "Show only confirmed")

Call GET /bookings?status=confirmed

Filter by date

Call GET /bookings?date=YYYY-MM-DD

Click a booking to view detail

Call GET /bookings/{booking_id} to render the full detail view (useful for a printable ticket or detail modal)

Cancel a booking button

Call DELETE /bookings/{booking_id}
On success, update the booking's status in UI state to "cancelled" without a full page reload


6. Print Flight / Seat Map (Requirement 01)
To display a flight with all seating marked

Call GET /flights/{flight_id} for the flight header info
Call GET /flights/{flight_id}/seats for the full seat grid
Render and trigger window.print() — the seat map with is_booked states gives you everything needed to show taken vs available seats visually


7. Admin Flows
Setting up a new aircraft type

POST /admin/aircraft — done once per new plane model

Generating flights for a new date (run daily, ideally via a scheduled job/cron)

For each of the 24 time slots → POST /admin/flights with { time_slot_id, aircraft_id, departure_date }
This creates the flight record and auto-generates all seat rows in one transaction
Best done the night before or early morning for the next operating day

Cancelling or taking a flight offline

PATCH /admin/flights/{flight_id} with { is_active: false }

Changing a gate

PATCH /admin/flights/{flight_id} with { gate: "C3" }

Printing the passenger manifest before boarding

GET /admin/flights/{flight_id}/manifest — returns all confirmed passengers with their seat codes, ordered by seat position


The complete happy path in one line:

Login → GET /slots → pick slot → GET /slots/{id}/flight → GET /flights/{id}/seats → pick seat → POST /bookings → done.
