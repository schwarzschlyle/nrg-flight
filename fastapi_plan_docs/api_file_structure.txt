---

## `app/main.py`
The FastAPI application entry point. Creates the `FastAPI()` instance, registers all custom exception handlers from `exceptions/handlers.py`, mounts the versioned API router from `api/main.py` under the `/api/v1` prefix, and configures startup/shutdown lifecycle events (e.g. opening and closing the database connection pool). This is the file your ASGI server (e.g. Uvicorn) points to.

---

## `app/models.py`
Contains all SQLAlchemy ORM model classes — one class per database table: `Account`, `Aircraft`, `TimeSlot`, `Flight`, `Seat`, `Booking`, `BookingStatus`. Each class maps directly to the schema defined earlier (UUIDs, `TIMESTAMPTZ`, FK relationships, constraints). This file is what Alembic reads to generate migrations. For a larger project, this becomes a `models/` directory with one file per domain.

---

## `app/crud.py`
Contains all raw database query functions — the only layer that talks directly to the database. Organized into logical groups:
- `get_account_by_email`, `create_account`
- `get_slots_by_date`, `get_flight_by_slot_and_date`, `get_flight_by_id`
- `get_seats_by_flight`, `get_first_available_seat`, `get_seat_by_code`
- `create_booking`, `get_booking_by_id`, `get_bookings_by_account`, `cancel_booking`
- `create_flight_with_seats` (transactional — inserts flight + all seat rows atomically)

All functions accept a database session as their first argument (injected via DI). No HTTP logic here — only DB operations. For a larger project, this becomes a `crud/` directory split by domain.

---

## `app/utils.py`
Stateless helper functions with no dependencies on the DB or HTTP layer:
- `generate_seat_codes(total_rows, seats_per_row)` — generates seat codes like "1A", "1B", …"30F"
- `hash_password(plain)` / `verify_password(plain, hashed)`
- `format_slot_label(slot_id)` — converts 0–23 to "00:00"–"23:00"
- `get_slot_flight_number(slot_id)` / `get_slot_gate(slot_id)` — returns the seeded flight number and gate for a given slot ID

---

## `app/api/main.py`
Aggregates all route modules into a single `APIRouter`. Imports routers from each file in `routes/` and includes them with their respective prefixes and tags:
- `/auth` → auth router
- `/flights` → flight router
- `/bookings` → booking router
- `/admin` → admin router

This router is then mounted in `app/main.py` under `/api/v1`.

---

## `app/api/deps.py`
All reusable FastAPI dependency functions used across routes via `Depends()`:
- `get_db()` — yields an async database session from the connection pool; ensures session is closed after the request
- `get_current_account()` — decodes and validates the JWT from the `Authorization` header; fetches and returns the account from DB; raises `401` if token is invalid
- `get_current_active_account()` — wraps `get_current_account()` and additionally checks `is_active = TRUE` and `deleted_at IS NULL`; raises `403` if inactive
- `require_admin()` — wraps `get_current_active_account()` and checks an admin claim in the JWT; raises `403` if not admin
- `get_pagination_params()` — extracts and validates `limit` and `offset` query params with defaults

---

## `app/api/routes/auth.py`
Handles all routes under `/auth`:
`POST /register`, `POST /login`, `POST /logout`, `POST /refresh`, `GET /me`.
Each endpoint uses schemas from `schemas/auth.py` for request/response validation. Calls `crud` functions for DB access. Uses `core/security.py` for password hashing and JWT creation. Raises domain exceptions from `exceptions/base.py` (e.g. `EmailAlreadyExistsError`, `InvalidCredentialsError`) rather than `HTTPException` directly.

---

## `app/api/routes/flight.py`
Handles all routes under `/flights` and `/slots`:
`GET /flights`, `GET /flights/{flight_id}`, `GET /slots`, `GET /slots/{slot_id}/flight`.
All read-only endpoints. Uses `get_current_active_account` dependency to protect routes. Returns Pydantic response schemas from `schemas/flight.py`. Raises `FlightNotFoundError` or `SlotNotFoundError` as appropriate.

---

## `app/api/routes/seat.py`
Handles all routes under `/flights/{flight_id}/seats`:
`GET /flights/{flight_id}/seats`, `GET /flights/{flight_id}/seats/first-available`, `GET /flights/{flight_id}/seats/{seat_code}`.
Protected by `get_current_active_account`. Returns `SeatResponse` schemas. Raises `SeatNotFoundError` when a seat code doesn't exist on a flight, `NoAvailableSeatsError` when the flight is fully booked.

---

## `app/api/routes/booking.py`
Handles all routes under `/bookings`:
`POST /bookings`, `GET /bookings`, `GET /bookings/{booking_id}`, `DELETE /bookings/{booking_id}`.
All routes protected by `get_current_active_account`. The `POST` route runs the transactional seat-locking logic via `crud.create_booking()`. Raises `SeatAlreadyBookedError` on conflict, `BookingNotFoundError`, and `BookingOwnershipError` (403) if the account doesn't own the booking.

---

## `app/api/routes/admin.py`
Handles all routes under `/admin`:
`POST /admin/aircraft`, `POST /admin/flights`, `PATCH /admin/flights/{flight_id}`, `GET /admin/flights/{flight_id}/manifest`.
All routes protected by `require_admin` dependency. `POST /admin/flights` calls the transactional `crud.create_flight_with_seats()` which generates all seat rows in one atomic operation.

---

## `app/core/config.py`
Loads and validates all environment variables using Pydantic's `BaseSettings`. Fields include:
`DATABASE_URL`, `SECRET_KEY`, `ACCESS_TOKEN_EXPIRE_MINUTES`, `REFRESH_TOKEN_EXPIRE_DAYS`, `ALGORITHM` (e.g. "HS256"), `ENVIRONMENT` ("development" | "production"). Raises a clear error on startup if any required variable is missing. Exported as a single `settings` singleton imported everywhere else.

---

## `app/core/security.py`
All authentication and cryptography logic:
- `hash_password(plain)` / `verify_password(plain, hashed)` — using `passlib`
- `create_access_token(data)` / `create_refresh_token(data)` — signs JWTs using `python-jose`
- `decode_token(token)` — verifies signature and expiry, returns payload
No HTTP logic here. Pure functions only.

---

## `app/core/db.py`
Sets up the async SQLAlchemy engine and session factory using `DATABASE_URL` from `config.py`. Exports:
- `engine` — the async database engine
- `AsyncSessionLocal` — the session factory used in `deps.get_db()`
- `Base` — the declarative base that all models in `models.py` inherit from

---

## `app/schemas/`
Pydantic models for request bodies and response shapes. Separated from ORM models so the DB layer and HTTP layer remain decoupled.

### `auth.py`
`RegisterRequest`, `LoginRequest`, `TokenResponse` (access_token, refresh_token), `RefreshRequest`, `AccountResponse` (id, email, full_name, has_verified_email)

### `flight.py`
`FlightResponse` (flight_id, flight_number, gate, departure_date, departure_time, aircraft_model, total_seats, available_seats), `SlotResponse` (slot_id, label, flight — nullable), `SlotFlightResponse`

### `seat.py`
`SeatResponse` (seat_id, seat_code, row_number, seat_letter, is_booked), `SeatMapResponse` (flight_id, seats: list[SeatResponse])

### `booking.py`
`CreateBookingRequest` (flight_id, seat_id), `BookingResponse` (booking_id, status, seat_code, flight_number, gate, departure_date, departure_time, booked_at), `BookingListResponse` (items: list[BookingResponse], total: int)

### `common.py`
Shared schema utilities: `PaginationParams`, `MessageResponse` (for simple `{"message": "..."}` responses)

---

## `app/exceptions/base.py`
All custom domain exception classes. None of these know anything about HTTP.
```
TicketingException (base)
├── EmailAlreadyExistsError
├── InvalidCredentialsError
├── AccountInactiveError
├── FlightNotFoundError
├── SlotNotFoundError
├── SeatNotFoundError
├── SeatAlreadyBookedError
├── NoAvailableSeatsError
├── BookingNotFoundError
└── BookingOwnershipError
```

---

## `app/exceptions/handlers.py`
Registers a `@app.exception_handler` for each custom exception class. Maps each domain exception to the appropriate HTTP status code and a consistent JSON error body `{"message": "..."}`. Imported and registered in `app/main.py` at startup. Keeps all HTTP error translation in one place, completely out of business logic.

---

## `app/alembic/versions/001_initial_schema.py`
The first Alembic migration. Contains `upgrade()` which runs the full `CREATE TABLE` SQL for all 7 tables in dependency order (lookup tables first, then tables with FKs), followed by seed inserts for `booking_status` and `time_slot`. Contains `downgrade()` which drops all tables in reverse order.

---

## `tests/conftest.py`
Pytest fixtures shared across all test files:
- `client` — a `TestClient` wrapping the FastAPI app
- `db_session` — a test database session (points to a test DB)
- `override_deps` — uses `app.dependency_overrides` to swap `get_db` with the test session
- `sample_account` / `sample_flight` / `sample_seat` — pre-built test data fixtures
- `auth_headers(account)` — returns `{"Authorization": "Bearer <test_token>"}` for authenticated test calls

---

## `tests/api/`
One test file per route module. Each file contains both unit tests (mocked dependencies) and integration tests (real test DB via overridden `get_db`):
- `test_auth.py` — register, login, token refresh, invalid credentials
- `test_flights.py` — slot listing, flight detail, date filtering
- `test_seats.py` — seat map rendering, first-available logic, seat code lookup
- `test_bookings.py` — happy path booking, double-booking conflict (409), cancellation and seat release, ownership enforcement (403)

---

## `.env` / `.env.example`
`.env` holds real secrets (gitignored). `.env.example` is committed to the repo with placeholder values so new developers know exactly which variables to configure:
```
DATABASE_URL=postgresql+asyncpg://user:password@localhost:5432/ticketing
SECRET_KEY=your-secret-key-here
ACCESS_TOKEN_EXPIRE_MINUTES=30
REFRESH_TOKEN_EXPIRE_DAYS=7
ALGORITHM=HS256
ENVIRONMENT=development
```

Schemas are separate from models. app/schemas/ holds Pydantic request/response shapes while app/models.py holds SQLAlchemy ORM classes. This keeps your DB layer and HTTP layer fully decoupled — a change in one doesn't force a change in the other.
Exceptions bubble up cleanly. crud.py and route files raise domain exceptions (SeatAlreadyBookedError, etc.) with zero knowledge of HTTP. exceptions/handlers.py is the single place that translates those into status codes and JSON responses — exactly as the article recommends.
deps.py is the DI hub. Every shared concern — DB sessions, auth, admin checks, pagination — lives as an injectable dependency. This makes unit testing trivial since you can swap any dependency via app.dependency_overrides in your test fixtures.
crud.py is the only DB-touching layer. Routes never write raw queries. This makes the data access layer replaceable and independently testable with mocked sessions.
core/ is infrastructure-only. Config, security crypto, and DB engine setup live here. No business logic, no HTTP concerns — just foundational plumbing that everything else depends on.
