Below is the **exact endpoint trigger flow** (in order) for the **user journey** + your 3 features.

> Base path: `{{baseUrl}}/api/v1`
> 
> **JWT header for protected endpoints:**
> `Authorization: Bearer <access_token>`

---

## 0) Preconditions (one-time per date/slot)
A user can only book seats if a **Flight** exists for `(departure_date, time_slot_id)`.

- If `GET /slots?date=...` returns `flight_id: null` for a slot, there is **no flight to book yet**.
- Admin must create it (once) using:
  1. `POST /admin/aircraft`
  2. `POST /admin/flights` (creates the flight + generates its seats)

---

## 1) User Authentication (JWT)
These are the first endpoints the frontend hits so you can call all protected endpoints.

1. **Register (optional if user already exists)**
   - `POST /auth/register`

2. **Login (required)**
   - `POST /auth/login`
   - Response includes `access_token`
   - Store `access_token` client-side and attach it as Bearer JWT for all steps below.

---

## 2) Date + Time Slot selection (24 hourly slots)
### 2.1 Show the 24 slots for a chosen date
3. **Fetch 24 slots**
   - `GET /slots?date=YYYY-MM-DD`  ✅ JWT required

   What you display per slot:
   - `slot_id`, `label` (e.g., `"13:00"`)
   - `flight_number`, `gate` (unique per slot)
   - `flight_id` (null means not created / not active)
   - `total_seats`, `available_seats`

   UI recommendation:
   - Disable a slot if `flight_id == null`.
   - Optionally mark as sold-out if `available_seats == 0`.

### 2.2 User clicks a specific slot
4. **Resolve the concrete flight for that slot**
   - `GET /slots/{slot_id}/flight?date=YYYY-MM-DD` ✅ JWT required

   Save:
   - `flight_id` (this is what the seat page uses)
   - `flight_number`, `gate`, `departure_time`

5. *(Optional but recommended)* **Fetch full flight details**
   - `GET /flights/{flight_id}` ✅ JWT required
   - Useful to show `total_seats` and `available_seats` at the top of the seat page.

---

## 3) Seat Selection Page (Feature 01)
### Feature 01 — Print / display plane seating with availability
6. **Fetch seat map for the flight**
   - `GET /flights/{flight_id}/seats` ✅ JWT required

The response contains `seats[]` where each seat has:
- `seat_id` (use this for booking)
- `seat_code` (e.g., `"1A"`)
- `is_booked` ✅ this is what the frontend uses to disable seats

Frontend behavior:
- Render a seat grid.
- Disable/gray-out seats where `is_booked == true`.

---

## 4) Auto-assign seat (Feature 02)
### Feature 02 — Assign the first available seat
7. **Get first available seat**
   - `GET /flights/{flight_id}/seats/first-available` ✅ JWT required
   - Returns `seat_id` + `seat_code`

8. **Book it**
   - `POST /bookings` ✅ JWT required
   - Body:
     ```json
     {
       "flight_id": "<flight_id>",
       "seat_id": "<seat_id>"
     }
     ```

9. *(Optional validation)* refresh the seat map to update UI:
   - `GET /flights/{flight_id}/seats`

Error handling:
- If no seats remain: `409 {"message":"No available seats"}`
- If someone booked it first: `409 {"message":"Seat already booked"}` → refresh seat map.

---

## 5) Book a specific seat code (Feature 03)
### Feature 03 — Assign a specific seat for a passenger (e.g., “1A”)
You have two valid flows:

### 5A) Preferred (UI already has seat_id from the seat map)
10. User clicks a seat button (you already know `seat_id` from step 6).
11. **Book it**
   - `POST /bookings` with `{flight_id, seat_id}`

### 5B) If user types a seat code (or you want server-side validation of a code)
10. **Resolve seat by code**
   - `GET /flights/{flight_id}/seats/{seat_code}` ✅ JWT required
   - Example: `GET /flights/{flight_id}/seats/1A`
   - Response includes `seat_id` and `is_booked`

11. **Book it**
   - `POST /bookings` with `{flight_id, seat_id}`

Error handling:
- Invalid seat code: `404 {"message":"Seat not found"}`
- Seat already taken: `409 {"message":"Seat already booked"}` → refresh seat map.

---

## 6) Booking confirmation + “My tickets”
After `POST /bookings`, you can show the confirmation from the response (it already includes `seat_code`, `flight_number`, `gate`, etc.).

Optional endpoints:
- `GET /bookings/{booking_id}` ✅ JWT required
- `GET /bookings` ✅ JWT required (supports `status`, `date`, `limit`, `offset`)

---

## Important note about “booked forever”
Right now the API includes:
- `DELETE /bookings/{booking_id}` which **cancels** the booking and **releases** the seat (`is_booked=false`).

If your requirement is “booked forever / cannot be undone”, tell me and I’ll adjust that behavior (e.g., remove cancel or make it not release the seat).
